# Quick Task 260703-246 — Summary

## Objective

Ensure AI song suggestions never draw from the deleted list, mirroring the existing hidden-song exclusion behavior.

## Key finding

In this codebase there is **no separate "deleted" field or collection** — soft-delete IS the `hidden` flag:

- `deleteSong()` (`src/stores/songs.ts`) sets `hidden: true`.
- The "Hidden Songs (N)" section in `SongsView.vue` is the deleted list; `restoreSong()` un-deletes via `hidden: false`.

Both AI entry points in `ServiceEditorView.vue` — `suggestAllSongs()` and `fetchAiForSlot()` — **already** filtered `songStore.songs.filter((s) => !s.hidden)` (labeled "D-18"). So deleted songs were already excluded functionally. This task hardens that guarantee.

## Changes

- **`src/stores/songs.ts`** — added `aiCandidateSongs` computed getter (excludes `song.hidden === true`), exported from the store's public API, placed after `allUserTags`. Single source of truth for the AI candidate pool.
- **`src/views/ServiceEditorView.vue`** — both AI entry points now derive `base` from `songStore.aiCandidateSongs` instead of duplicating the inline `songStore.songs.filter((s) => !s.hidden)` filter. Orchestra sub-filtering and the D-18 comments left untouched.
- **`src/stores/__tests__/songs.test.ts`** — added `describe('aiCandidateSongs — soft-deleted exclusion', ...)` with 3 cases: excludes `hidden: true`, includes `hidden: false`, includes legacy docs with `hidden` undefined.

## Verification

- `npx vue-tsc --build --force` — clean.
- `npx vitest run src/stores/__tests__/songs.test.ts` — 66/66 passing.
- Grep confirmed no remaining inline `songStore.songs.filter((s) => !s.hidden)` in `ServiceEditorView.vue`.

## Result

Deleted (soft-deleted / hidden) songs are guaranteed to stay out of the AI suggestion candidate set via a single, tested store getter — no duplicated inline filters left to drift.
