<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| unauthenticated browser → serviceShares doc (read) | Public, anonymous read of a denormalized share doc. No auth token present. |
| authenticated editor → serviceShares doc (write) | Editor of the embedded orgId writes/overwrites a deterministic-id doc. |
| any authenticated user → another org's serviceShares doc | Cross-org attacker with a valid account guesses the `{slug}__service-{date}` id and attempts to overwrite. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-17-02-01 | Tampering | serviceShares create/update (guessable doc id) | high | mitigate | `allow create/update: if isOrgEditor(request.resource.data.orgId)` + `request.resource.data.orgId == resource.data.orgId` on update; proven by rules.test.ts cross-org create/update/orgId-change-fail cases and enforced live via the deploy checkpoint. |
| T-17-02-02 | Elevation of Privilege | non-editor / cross-org overwrite | high | mitigate | Editor-scoped write rule; emulator tests assert non-member and cross-org writes fail. |
| T-17-02-03 | Information Disclosure | public read of serviceShares | low | accept | Public read is by design (mirrors quarterShares). The doc will carry only names (enforced in 17-03), never email/phone; anonymous read of names-only schedule data is the intended feature. |
| T-17-02-04 | Spoofing | slug squatting on 'service-share' | low | mitigate | `'service-share'` added to RESERVED_SLUGS so no org can claim it as its memorable slug. |
| T-17-02-SC | Tampering | npm/pip/cargo installs | n/a | accept | No package installs this phase. |
</threat_model>
