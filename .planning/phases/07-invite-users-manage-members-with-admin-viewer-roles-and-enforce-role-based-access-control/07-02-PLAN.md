Create `src/views/TeamView.vue` — a full-page team management view accessible at `/team` (editor-only, per Plan 01 router guard).

**Page structure (top to bottom):**

1. **Wrap in AppShell** (established pattern — import AppShell, wrap template content)

2. **Page header:**
   - Title: "Team" (text-xl font-semibold text-gray-100)
   - Subtitle: org name from authStore.orgName (text-sm text-gray-400)

3. **Inline invite form** (above the member table):
   - Email input (type="email", placeholder="Enter email address", bg-gray-800 border-gray-700 text-gray-100 rounded-md px-3 py-2 text-sm)
   - Role select dropdown with two options: "Editor" and "Viewer" (same styling, default "viewer")
   - "Invite" button (bg-indigo-600 hover:bg-indigo-500 text-white rounded-md px-4 py-2 text-sm font-medium)
   - Error message area (text-red-400 text-sm mt-1) for validation errors
   - Success feedback: brief "Invited!" text that fades after 2 seconds (use a ref + setTimeout, same pattern as pcCopied in ServiceEditorView)

4. **Invite validation (before writing):**
   - Email must be non-empty and pass basic format check (includes '@' and '.') — show "Enter a valid email address"
   - Email must NOT match any existing member's email (compare lowercase) — show "This person is already a member"
   - Email must NOT match any pending invite email — show "An invite has already been sent to this email"

5. **Invite creation logic:**
   - Normalize email: `email.trim().toLowerCase()`
   - Use `writeBatch(db)` to write TWO docs atomically:
     a. `organizations/{authStore.orgId}/invites/{normalizedEmail}` with `{ role: selectedRole, invitedBy: authStore.user.uid, invitedAt: serverTimestamp(), email: normalizedEmail, status: 'pending' }`
     b. `inviteLookup/{normalizedEmail}` with `{ orgId: authStore.orgId, role: selectedRole, invitedAt: serverTimestamp() }`
   - After batch commit: clear email input, show "Invited!" feedback

6. **Member table** (rounded-lg border border-gray-800 overflow-hidden, matching SongTable pattern):
   - Columns: Name, Email, Role, Joined, Actions
   - Header row: bg-gray-800/50 border-b border-gray-700, text-xs font-medium text-gray-400 uppercase tracking-wider
   - Body rows: divide-y divide-gray-800
   - **Pending invite rows** (from invites subscription):
     - Name column: show email (text-gray-400)
     - Email column: same email
     - Role column: role badge (Editor = blue, Viewer = gray) + "Pending" badge (bg-yellow-900/30 text-yellow-400 px-1.5 py-0.5 text-xs rounded)
     - Joined column: "Invited" text
     - Actions: "Cancel" button to delete the invite (deletes both invites/{email} and inviteLookup/{email} via writeBatch)
   - **Active member rows** (from members subscription):
     - Name column: displayName or email prefix (text-gray-200)
     - Email column: email (text-gray-400)
     - Role column: badge — Editor (bg-indigo-900/50 text-indigo-300) or Viewer (bg-gray-700 text-gray-300)
     - Joined column: formatted date (text-gray-400 text-sm)
     - Actions column (only show actions for members that are NOT the current user):
       - Role toggle button: "Make Viewer" or "Make Editor" (text-sm text-gray-400 hover:text-gray-200)
       - Remove button: "Remove" (text-sm text-red-400 hover:text-red-300)
       - **Guard for current user row:** Show "You" label instead of action buttons
       - **Last editor guard:** Before removing or demoting the last editor, check `members.filter(m => m.role === 'editor').length`. If count is 1 and target is current user or the only editor, show error "Cannot remove the only editor. Assign another editor first." and block the action.

7. **Data subscriptions (onMounted):**
   - Subscribe to `organizations/{authStore.orgId}/members` via `onSnapshot(collection(...))` — map docs to `{ uid: doc.id, ...doc.data() }` array
   - Subscribe to `organizations/{authStore.orgId}/invites` via `onSnapshot(collection(...))` — map docs to `{ email: doc.id, ...doc.data() }` array
   - Use `authStore.orgId` directly (do NOT fetch orgId from users/{uid})
   - Guard: if `!authStore.orgId`, show a loading state and return
   - Clean up both subscriptions in `onUnmounted`

8. **Role change action:**
   - `updateDoc(doc(db, 'organizations', authStore.orgId, 'members', targetUid), { role: newRole })`
   - No confirmation dialog needed for role toggle (instant feedback via onSnapshot)

9. **Remove member action:**
   - Show a simple inline confirm: "Remove {name}?" with Confirm/Cancel buttons (toggle a `confirmingRemoveUid` ref)
   - On confirm: `deleteDoc(doc(db, 'organizations', authStore.orgId, 'members', targetUid))`
   - Also update `users/{targetUid}` to remove the orgId from orgIds array? No — we cannot read/write other users' docs due to security rules. The removed user will simply get a permission-denied on their next Firestore access, and their auth store's member snapshot listener will fire with `exists() = false`, setting userRole to null. This effectively locks them out.

10. **Imports needed:** AppShell, useAuthStore, firebase db, doc, collection, onSnapshot, writeBatch, setDoc, deleteDoc, updateDoc, serverTimestamp, ref, computed, onMounted, onUnmounted
