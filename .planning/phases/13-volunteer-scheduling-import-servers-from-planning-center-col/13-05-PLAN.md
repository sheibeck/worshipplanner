<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser client → Firestore | Roster PII (names, emails, phones) written/read under organizations/{orgId} |
| Cross-org | A signed-in user must only read/write their own org's roster |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-05-01 | Information Disclosure | Cross-org roster PII read/write | mitigate | All reads/writes scoped to `organizations/{orgId}/...`; the existing `firestore.rules` catch-all `match /{collection}/{docId}` under org already enforces `isOrgEditor(orgId)` — new `people`/`roles` collections inherit it (RESEARCH V4). No rules change required; confirm no code path uses a raw top-level collection |
| T-13-05-02 | Tampering | Re-import silently overwriting a deactivated person back to active | mitigate | upsertPeople excludes `active` from the update payload on match — a deactivated volunteer re-imported stays inactive (test-enforced) |
| T-13-05-03 | Information Disclosure | Roster PII exposed to viewer-role members | mitigate | Roster is editor-only by design (mirrors `songs`, stricter than `services`); router meta requiresEditor (Plan 07) + the org editor Firestore rule gate both apply. No viewer read path is created |
| T-13-05-04 | Tampering | Blackout/pairing data corrupting standing roster fields on quarter reset | mitigate | Person doc holds ONLY standing fields; quarter-scoped data lives in the quarter doc (Plan 06) — grep gate confirms no blackout/pairing writes here (RESEARCH Pitfall 3) |
</threat_model>
