<objective>
Migrate existing Phase 13/14 data to the per-role shape without a batch script, and persist the new `roleFrequencies` map through the roster store's write path — TDD-first.

Purpose: D-03 requires each existing person's tuned single cadence to be copied onto all roles they currently hold (nothing already tuned is lost); D-09 requires the seeded `vocals` role to reclassify from group `band` to `vocals`. The ONLY migration precedent in this repo is `auth.ts`'s opportunistic patch-on-read inside an `onSnapshot` handler — guarded and idempotent. This plan follows that exact pattern (no Admin SDK, no scripts/ dir). The CSV import path must keep working by degrading gracefully: one `Frequency` column applies to every role in the row (Pitfall 4 — no CSV schema change).

Output: guarded, idempotent migrations in roster.ts's people + roles snapshot handlers; roleFrequencies wired through upsertPeople/updatePerson/addPerson with graceful-degrade defaulting; unit tests via the existing mocked-Firestore harness.
</objective>

    D-04 persistence: upsertPeople/updatePerson/addPerson round-trip roleFrequencies.
    D-07/D-02 CSV graceful degrade: when a caller supplies roles but no roleFrequencies, the store applies the person's frequencyTargetN (or N=4 default when absent) to every role in `roles` as the initial roleFrequencies map — mirrors D-03 at import time.
  </behavior>

[]).map(r => [r, data.frequencyTargetN ?? 4]))` and `await updateDoc(d.ref, { roleFrequencies })`. Guard on absence only (idempotent). In the upsert write-path, when `incoming.roleFrequencies !== undefined` set it; additionally, when `incoming.roles` is provided but `incoming.roleFrequencies` is not, synthesize the map from `incoming.roles` × (`incoming.frequencyTargetN ?? existing?.frequencyTargetN ?? 4`) so CSV import stays compatible without a schema change. Keep the existing roles-merge (never replace) semantics. Do NOT touch blackoutDates/pairedWith (D-07 — those are quarter-scoped, in quarters.ts, out of this plan).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Firestore snapshot → migration patch | Untrusted/legacy doc shapes drive `updateDoc` patches back to Firestore |
| Caller (CSV/manual) → upsert write | Caller-supplied person input crosses into a scoped Firestore write |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-03-01 | Tampering | D-03 migration overwriting already-tuned cadences | mitigate | Guard fires ONLY when `roleFrequencies` is wholly absent; idempotency asserted by a test that confirms no second patch when the field exists. Existing tuned values are never clobbered. |
| T-15-03-02 | Tampering | Migration patch clobbering sibling person data or other docs | mitigate | Patch targets each doc's own `ref` with a single-key object (`{ roleFrequencies }` / `{ group }`) — no read-modify-write of the whole doc, no cross-doc writes. Mirrors auth.ts dot-scoped precedent. |
| T-15-03-03 | Elevation of Privilege / Integrity | Invalid cadence value written to Firestore | mitigate | Synthesized N derives from validated `frequencyTargetN` (already clamped upstream) or the constant default 4; UI-supplied cadence continues to use the existing `[1,8]` clamp (RESEARCH Security Domain V5). No unbounded/negative N introduced. |
| T-15-03-SC | Tampering | package installs | accept | No packages installed this phase (RESEARCH audit N/A). |
</threat_model>
