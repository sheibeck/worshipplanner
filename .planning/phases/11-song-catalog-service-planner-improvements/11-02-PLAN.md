    Three interlocking fixes — implement together because D-15's immediate save is also part of D-17's cure.

    **D-16 snap-back (root cause: SortableJS physically moves the DOM node, then Vue re-renders from the reactive array; the two orderings fight).** In the `onEnd(evt)` handler (ServiceEditorView.vue:974-995), BEFORE mutating `localService.value.slots`, revert SortableJS's DOM mutation so Vue is the single source of truth:
    ```typescript
    onEnd(evt) {
      if (!localService.value || evt.oldIndex == null || evt.newIndex == null) return
      if (evt.oldIndex === evt.newIndex) return
      // Revert Sortable's DOM move so Vue's reactive render is authoritative (prevents snap-back)
      const parent = evt.item.parentNode
      if (parent) {
        const ref = parent.children[evt.oldIndex]
        parent.insertBefore(evt.item, evt.oldIndex < evt.newIndex ? ref?.nextSibling ?? null : ref ?? null)
      }
      const slots = [...localService.value.slots]
      const moved = slots.splice(evt.oldIndex, 1)[0]
      if (!moved) return
      slots.splice(evt.newIndex, 0, moved)
      const reindexed = reindexSlots(slots)
      localService.value.slots = reindexed
      // D-15 immediate persist (below)
    }
    ```
    Also change the v-for `:key` (ServiceEditorView.vue:456) from the index-dependent `slot.position + '-' + slot.kind + '-' + index` to a stable per-slot identity that does not shift on reorder (e.g. a stable `slot.id` if one exists, otherwise `slot.kind + '-' + slot.position` WITHOUT the array index) so Vue reconciliation matches moved nodes correctly. Do NOT keep `index` in the key.

    **D-15 immediate save.** Inside `onEnd`, after `localService.value.slots = reindexed`, persist immediately rather than waiting on the 800ms debounce:
    ```typescript
    if (serviceId.value) {

      try {
        await serviceStore.updateService(serviceId.value, { slots: reindexed })
        originalService.value = JSON.parse(JSON.stringify(localService.value))
        autosaveStatus.value = 'saved'
      } finally {
        autosaveSaving = false
      }
    }
    ```

If a remote merge is skipped because status is pending/saving, and after it settles `isDirty` is still true, the autosave watcher's re-arm guard above must pick it up.
    - When wiring the D-15 immediate save, set `autosaveSaving`/`autosaveStatus` consistently (as shown) so the inflight guard cannot deadlock (`autosaveSaving` always reset in `finally`).

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-11-05 | Tampering | ScriptureInput preview-close vs delete X | mitigate | Preview-close handler isolated (`@click.stop`, distinct button) so dismissing a preview cannot delete the slot (D-13) |
