<task type="auto" tdd="true">
  <name>Task 1: Service Pinia store with Firestore subscription and CRUD</name>
  <files>src/stores/services.ts, src/stores/__tests__/services.test.ts</files>
  <behavior>
    - subscribe(orgId) sets up onSnapshot on organizations/{orgId}/services ordered by date desc
    - unsubscribeAll() cleans up listener and resets state
    - createService(data) writes to Firestore with serverTimestamp for createdAt/updatedAt, returns doc id
    - updateService(id, data) updates document with serverTimestamp for updatedAt
    - deleteService(id) deletes document
    - assignSongToSlot(serviceId, slotPosition, song) updates the specific slot in the service's slots array AND calls useSongStore().updateSong(songId, { lastUsedAt: serverTimestamp() })
    - clearSongFromSlot(serviceId, slotPosition) clears songId/songTitle/songKey back to null on the slot
    - services ref starts as empty array, isLoading starts as true
    - After onSnapshot callback fires, isLoading becomes false
  </behavior>
  <action>
    Create `src/stores/services.ts` following the exact pattern from `src/stores/songs.ts`:

    1. Setup store with `defineStore('services', () => { ... })`
    2. State: `services = ref<Service[]>([])`, `isLoading = ref(true)`, `orgId = ref<string | null>(null)`
    3. `subscribe(orgIdValue)`: unsubscribe previous, set orgId, create query on `organizations/{orgId}/services` ordered by `date` desc, set up `onSnapshot` that maps docs to Service[]
    4. `unsubscribeAll()`: cleanup listener, reset state
    5. `createService(data: Omit<ServiceInput, 'slots'>  & { progression: Progression })`:
       - Import `buildSlots` from `@/utils/slotTypes`
       - Build the 9-slot template from the progression
       - Write to Firestore: `{ ...data, slots: buildSlots(data.progression), status: 'draft', notes: '', sermonPassage: null, createdAt: serverTimestamp(), updatedAt: serverTimestamp() }`
       - Return the document id
    6. `updateService(id, data)`: update document with serverTimestamp for updatedAt
    7. `deleteService(id)`: delete document
    8. `assignSongToSlot(serviceId, slotPosition, song: { id: string, title: string, key: string })`:
       - Read the current service from `services.value` to get current slots array
       - Clone slots, update the slot at `slotPosition` with `songId: song.id, songTitle: song.title, songKey: song.key`
       - Write updated slots to Firestore via `updateService`
       - CRITICAL cross-store write: call `useSongStore().updateSong(song.id, { lastUsedAt: serverTimestamp() })` to track usage — this is the key link between service planning and the suggestion algorithm
    9. `clearSongFromSlot(serviceId, slotPosition)`: set songId/songTitle/songKey back to null on the slot

    Write tests in `src/stores/__tests__/services.test.ts` following the mocking pattern from `src/stores/__tests__/songs.test.ts`. Mock `firebase/firestore` and `@/firebase` modules. Test createService builds correct slots, assignSongToSlot calls updateSong on the song store.
  </action>
  <verify>
    <automated>npx vitest run src/stores/__tests__/services.test.ts --reporter=verbose</automated>
  </verify>
  <done>Service store creates services with correct slot templates, assigns songs to slots, writes lastUsedAt back to song store. All tests pass.</done>
</task>
