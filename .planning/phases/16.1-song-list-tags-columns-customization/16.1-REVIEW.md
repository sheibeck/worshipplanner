`removedThemes` with a real value, making the feature dead on arrival for end users.

## Critical Issues

### CR-01: VW-type filter is not cleared/gated when VW mode is turned off, silently hiding songs with no visible cause

**Resolved:** `b4b78c7` — `matchesVwType` in `songs.ts` is now gated on `!authStore.vwModeEnabled ||
...`, mirroring the existing `matchesSearch` gate. Tests added in `songs.test.ts` assert the
dropdown filter is inert (both numeric and `'uncategorized'`) when VW mode is disabled. This
also resolves WR-03 (dashboard "Uncategorized" tile symptom), which was flagged as a secondary
symptom of this same root cause.

**File:** `src/stores/songs.ts:60-64` (also `src/components/SongFilters.vue:47-58`)

**Issue:** `filteredSongs` computed gates the search bar's `type:` prefix on `authStore.vwModeEnabled`
(line 58, correct), but the separate `filterVwType` dropdown filter is applied unconditionally:

```ts
const matchesVwType =
  filterVwType.value === null ||
  (filterVwType.value === 'uncategorized'
    ? song.vwTypes.length === 0
    : song.vwTypes.includes(filterVwType.value as VWType))
```

`SongFilters.vue` hides the `<select>` control entirely when VW mode is off
(`v-if="authStore.vwModeEnabled"`, line 48), but never resets `songStore.filterVwType` to `null`
when that happens. If a user selects "Type 2" and later an editor disables VW mode in Settings
(`SettingsView.vue`), the Songs list keeps silently excluding every song whose `vwTypes` doesn't
include `2` — with the only control that could explain or clear this filter now gone from the DOM.
This directly contradicts this phase's own D-16 requirement ("hide all VW indicators app-wide —
**visual hide only, no data deletion**"): the filter's effect on visible data is not merely hidden,
it becomes invisible-but-active, and there is no UI path to discover or undo it short of
re-enabling VW mode.

**Fix:** Reset `filterVwType` (and ideally re-validate it) whenever `vwModeEnabled` flips to
`false`. E.g. in `songs.ts`, watch `authStore.vwModeEnabled` (or gate `matchesVwType` on it,
mirroring the `matchesSearch` gate):

```ts
const matchesVwType =
  !authStore.vwModeEnabled ||
  filterVwType.value === null ||
  (filterVwType.value === 'uncategorized'
    ? song.vwTypes.length === 0
    : song.vwTypes.includes(filterVwType.value as VWType))
```

or clear `songStore.filterVwType = null` directly inside `SettingsView.vue`'s `onToggleVwMode()`
when `newValue === false`.

### CR-02: D-14 "removed theme doesn't reappear on re-import" has no write path anywhere in the shipped UI

**File:** `src/components/SongSlideOver.vue:440-468`, `src/components/SongTable.vue:419-422`

**Issue:** The store-layer logic for D-14 is correct and tested
(`songs.ts` `upsertSongs` subtracts `existing.removedThemes` from the incoming theme union, and
preserves `removedThemes` verbatim across re-import — see
`src/stores/__tests__/songs.test.ts:1104-1146`). However, nothing in the final shipped UI ever
writes a non-empty value into `removedThemes`:

- `SongSlideOver.vue`'s Themes field is a single free-text comma-separated re-entry
  (`themesInput`, lines 126-137). When a user removes a theme from that text and clicks Save, the
  new (shorter) `themes` array is written, but `removedThemes` is passed through unchanged
  (`removedThemes: props.song?.removedThemes ?? []`, line 467) — the removal is never recorded.
- `SongTable.vue`'s Themes column is now display-only + click-to-filter (confirmed by its own
  test asserting `button[aria-label="Remove theme"]` no longer exists — see
  `src/components/__tests__/SongTable.test.ts:143-145`). The inline add/remove-with-`removedThemes`
  wiring that plan 05's summary described was subsequently removed in a later wave, but the code
  comment left behind is now factually wrong:

  ```ts
  // Note: Tags/Themes on this listing are display-only + click-to-filter
  // (filterByPill above). All add/edit/remove of tags and themes — including the
  // removedThemes tracking (D-14) that survives PC re-import — lives on the edit
  // screen (SongSlideOver.vue), not here.
  ```

  `SongSlideOver.vue` does not implement any such tracking (see above) — the comment describes
  functionality that does not exist in this codebase.

Net effect: a user has no way, anywhere in the app, to remove a theme such that it "does NOT
reappear after the next PC import" — the exact acceptance criterion stated in plan 01
(`16.1-01-PLAN.md:20`). Any theme removed via `SongSlideOver` will resurface the next time Planning
Center is re-imported, because `removedThemes` never gets the removed value appended.

**Fix:** Either (a) restore an explicit per-theme remove affordance somewhere in the UI (e.g. back
in `SongSlideOver.vue`'s Themes editor, converting it from a single text field to individual
removable pills + free-text add, mirroring the User Tags pattern already in that same file) that
computes `removedThemes` as `existing removedThemes ∪ (old themes − new themes)` on save, or (b) if
per-theme removal tracking is being deliberately dropped as a feature, update `Song.removedThemes`'s
usage/docs accordingly and correct/remove the misleading comment in `SongTable.vue:419-422`. As
shipped, the plumbing exists but is unreachable, which is worse than either alternative — it gives
false confidence that D-14 is handled.

**Resolved:** `21da925` — took option (a). `SongSlideOver.vue`'s `onSave` now computes
`oldThemes − newThemes` against the themes captured at edit-open time, unions the result into
`props.song?.removedThemes ?? []` (de-duped), and prunes any theme that's present again in the
saved `themes` array (re-adding un-removes it). The misleading comment in `SongTable.vue:419-422`
was corrected to state that this tracking lives in `SongSlideOver.vue` and is recorded on save.
Tests added in `src/components/__tests__/SongSlideOver.test.ts`.

## Warnings

### WR-01: Click-to-filter de-dupe breaks for multi-word tag/theme values

**File:** `src/components/SongTable.vue:361-372`

**Issue:** `filterByPill` de-dupes by splitting the current search string on whitespace and
checking whether the exact term is already a token:

```ts
function filterByPill(field: 'type' | 'tag' | 'theme', value: string | number) {
  const term = `${field}:${value}`
  const current = songStore.searchQuery.trim()
  if (!current) {
    songStore.searchQuery = term
    return
  }
  const alreadyPresent = current.split(/\s+/).includes(term)
  if (alreadyPresent) return
  songStore.searchQuery = `${current} ${term}`
}
```

For a tag/theme value containing a space (a valid input — `SongSlideOver.vue`'s free-text tag
adder only splits on commas, so `"Christmas Eve"` is a perfectly valid single tag), `term` itself
contains a space (e.g. `"tag:Christmas Eve"`). `current.split(/\s+/)` breaks any previously-appended
occurrence of that same term into separate single-word tokens (`"tag:Christmas"`, `"Eve"`), neither
of which equals the multi-word `term`, so `alreadyPresent` is always `false` for such values.
Clicking the same multi-word pill twice appends a second, duplicate `tag:Christmas Eve` span to the
search box instead of no-op'ing as the surrounding comment promises ("re-clicking the same pill is
a no-op (de-duped)"). Not caught by `SongTable.test.ts` (no multi-word-tag click test exists).

**Fix:** De-dupe by checking whether `term` appears as a whole field-scoped span rather than as a
single whitespace token, e.g. reuse the same `FIELD_SPAN_RE`-style extraction from `songSearch.ts`,
or simplest: check `current === term || current.startsWith(term + ' ') || current.includes(' ' + term + ' ') || current.endsWith(' ' + term)`.

**Resolved:** `1b42493` — `filterByPill`'s de-dupe now checks the exact suggested whole-span
condition instead of splitting on whitespace. Multi-word pill click tests added in
`src/components/__tests__/SongTable.test.ts`.

### WR-02: Settings' VW-mode toggle bypasses the auth store's action pattern and doesn't cascade the reset from CR-01

**File:** `src/views/SettingsView.vue:474-499`

**Issue:** Every other cross-cutting org-level auth field (`pcAppId`/`pcSecret`) is mutated through
a dedicated store action (`setPcCredentials`). `onToggleVwMode` instead reaches into the store and
assigns the ref directly: `authStore.vwModeEnabled = newValue`. This is not incorrect for a Pinia
setup store (the ref is genuinely mutable from outside), but it's an inconsistent pattern within
the same file/store, and it's also the reason CR-01 has no natural place to cascade a reset from —
there's no `setVwModeEnabled` action for `songStore.filterVwType` to hook a watcher/side-effect off
of.

**Fix:** Add a `setVwModeEnabled(value: boolean)` action on `useAuthStore` (mirroring
`setPcCredentials`) and call it from `SettingsView.vue`. Use that as the single choke point to also
clear `songStore.filterVwType` (see CR-01's fix) when VW mode is disabled.

**Resolved:** `b4b78c7` (side effect of CR-01's fix) — `filterVwType`'s `'uncategorized'` value is

### IN-02: `upsertSongs`'s grow-only tag union can resurrect a user-removed tag on PC re-import (documented tradeoff, flagged for visibility)

**File:** `src/stores/songs.ts:360-364`

**Issue:** Unlike themes (which have `removedThemes` tracking, even though currently unreachable —
see CR-02), a user-removed *tag* has no removal-tracking mechanism at all: `upsertSongs` always
unions `existing.tags` with incoming PC tags, so a tag a user explicitly deleted via
`SongSlideOver.vue`'s `removeUserTag` will reappear on the next PC re-import if PC still reports it
as a team-style tag. This is explicitly called out in the code's own comment
(`songs.ts:360-363`, "Tradeoff: a user-removed *tag*... can reappear on reimport") as an accepted
design tradeoff, not an oversight, so it is not filed as a Warning — surfaced here only for
completeness given the review's explicit focus on tag-fold data-loss/reappearance scenarios.

**Fix:** N/A — documented and accepted behavior. Consider extending `removedThemes`-style tracking
to tags in a future phase if this proves confusing in practice.
