# Phase 16.1: Song list tags & columns customization - Context

**Gathered:** 2026-07-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Simplify and make customizable the Songs screen's tag/theme/column display, and make the app's Vertical Worship (VW) 1-2-3 conventions self-explanatory and optional. Four scoped items (from ROADMAP.md R-01..R-04):

1. **R-01 — Fold team tags into Tags.** Remove the separate "Team" tag concept on songs; songs carry one flat tag set. Decouple songs from service teams.
2. **R-02 — Themes as a separate column.** Split Themes out of the combined Tags cell into its own song-list column.
3. **R-03 — Column-visibility settings cog.** A per-user control to show/hide individual song-list columns.
4. **R-04 — Explain (and make optional) the VW 1-2-3 methodology.** Add an in-app explainer of the 1-2-3 categories, and make the whole VW paradigm an optional package toggled in Settings.

This clarifies HOW to implement the four items. The one deliberate expansion beyond a pure Songs-screen change is R-04's "make optional" — the user chose to build a Settings toggle now that hides VW indicators app-wide (R-04 explicitly contemplated optionality). No other new capabilities.

</domain>

<decisions>
## Implementation Decisions

### R-01 — Fold team tags into one flat tag set + decouple songs from teams
- **D-01:** Merge `Song.teamTags` into the existing flat `tags` array and **remove the dedicated `teamTags` field** from `Song` (and drop its special-case handling). Migrate existing `teamTags` values into `tags`, de-duplicated. Songs have exactly ONE set of tags.
- **D-02:** The **service-level team scheduling** concept (which teams — Choir, Orchestra, Special Service — play on a given Sunday) is **unchanged**. It stays a property of the service. Only the song↔team *link* is severed; songs are no longer intrinsically bound to a team.
- **D-03:** Remove the **hard team filter** in `rankSongsForSlot` (`src/utils/suggestions.ts`) — Choir/Special-Service no longer gate which songs are eligible. All songs are always eligible.
- **D-04:** Keep a **soft nudge only**: generalize today's Orchestra +200 bonus so that any song whose flat `tags` include a **scheduled team's name** ranks higher (string match, case-insensitive, data-driven — no hardcoded team list). Orchestra is the primary case; Choir is a likely future case ("big choir numbers we tag as Choir"). Songs always appear regardless of the nudge.
- **D-05:** PC import that previously mapped arrangement titles (e.g. Orchestra) → `teamTags` now writes those into `tags` instead, preserving the same import-safety/merge protection already applied to user `tags` and `hidden` (never clobber user-added tags on re-import).
- **D-06:** The `team:` search prefix (Phase 12 D-03) becomes redundant once `teamTags` is gone. Fold it away or alias it to a plain tag match — planner's choice — so folded team tags remain findable via `tag:`/bare-term search. Update `songSearch.ts` + tests accordingly.

### R-02 / R-03 — Themes column + column-visibility cog
- **D-07:** Add **Themes as its own song-list column** in `SongTable.vue`, separate from the Tags column.
- **D-08:** A **settings-cog control on the song list** lets the user show/hide individual columns. **Title is always visible** (row identity + edit affordance); every other column — Category, Key, CCLI, Last Used, Tags, Themes — is toggleable.
- **D-09:** **Default = all columns visible** (including the new Themes column).
- **D-10:** Column-visibility choices persist **per-user via localStorage**, reusing the Phase 12 per-user/org key pattern (`src/stores/songs.ts` localStorage helpers). This is a **personal view preference**, distinct from the church-level VW setting (D-15).
- **D-11:** Column customization applies to the **Songs-management panel (`SongTable`) only**. The service-plan song picker (`SongSlotPicker.vue`) keeps its own compact layout, untouched.

### R-02 (cont.) — Tags vs Themes column behavior
- **D-12:** Two distinct columns render: **Themes** (PC-imported thematic keywords) and **Tags** (the single flat set = folded team tags + user tags).
- **D-13:** **Both columns are inline-editable** (add/remove pills inline, mirroring today's user-tag inline edit). Themes are no longer read-only.
- **D-14:** Apply the existing **import-preservation semantics** to edited themes (Phase 11 D-08 merge-on-reimport). Known nuance to handle: a locally-*removed* theme that still exists in Planning Center may reappear on the next PC import unless removals are tracked — planner decides whether to track removals or accept reappearance; document the chosen behavior.

### R-04 — VW 1-2-3 optional package + explainer
- **D-15:** VW 1-2-3 becomes an **optional package toggled in Settings**, **default ON**. Stored as a **church/account-level setting** (it defines the shared planning methodology for the team), NOT a per-user localStorage flag. Researcher confirms exactly where existing app settings persist today (Settings view / Firestore org/account doc — cf. Phase 8 PC credentials, Phase 16 slug field).
- **D-16:** When VW mode is **OFF**, hide all VW indicators **app-wide** (visual hide only — no data deletion): the Category column + VW badge in the song list AND the service-plan picker, the 1-2-3 slot typing/badges in the service editor, the `type:` search prefix and the VW-type filter dropdown, and any VW rotation/progression hints. The underlying `vwTypes` data is retained so toggling back ON restores everything.
- **D-17:** When VW mode is **ON**, provide an **in-app explainer** describing the app's actual **1-2-3** categories: **1 = Call to Worship, 2 = Intimate, 3 = Ascription**, framed with the horizontal→vertical "Gather / Testimony / Arrive" philosophy and the goal of ending on a 3 (pure ascription/adoration). Claude drafts the copy for the user's review during planning/execution. Explainer placement (help icon / popover near the Category column / Settings help link) is Claude's discretion, following existing dark-mode UI patterns.
- **D-18:** The explainer teaches the app's **1-2-3** model as-is. Do **NOT** expand to the full 1-to-5 "Gather/Testimony/Arrive" scale from the source text — that would change `VWType` (1|2|3), badges, suggestions, and progressions and belongs in its own future phase. See Deferred Ideas.

### Claude's Discretion
- Exact soft-nudge scoring value/curve in `rankSongsForSlot` (generalize the existing +200 Orchestra pattern), as long as D-03/D-04 semantics hold (no hard filter; songs always appear; team-name↔tag match boosts).
- Settings-cog UI affordance (popover vs dropdown vs inline panel) and its placement on the Songs panel — follow `SongFilters.vue` conventions and the gray-950/900/800 palette.
- localStorage key naming/versioning for column visibility (mirror Phase 12 D-13 per-user/org derivation).
- Whether the `team:` search prefix is deleted or aliased to plain tag matching (D-06).
- Exact explainer copy (draft for review) and placement/affordance (D-17).
- How themes-removal-vs-reimport is reconciled (track removals vs accept reappearance) (D-14).
- Whether to keep or drop the now-redundant teamTags-specific branches in the unified tag-filter checklist (Phase 12 D-08 union was `teamTags ∪ themes ∪ tags`; after the merge it is just `themes ∪ tags`).

</decisions>

- `TeamTagPill.vue` variants (`team`/`theme`) — the theme variant stays; the team variant is absorbed into ordinary tag styling.

- Pinia stores with Firestore `onSnapshot`; computed `filteredSongs`; client-side sort in `SongTable`.
- Search: multi-term AND + field-scoped prefixes in `songSearch.ts` (Phase 12).
- Import upsert preserves user-controlled fields (`tags`, `hidden`) across re-import — extend to folded team tags + edited themes (D-05/D-14).
- Settings persisted at the church/account level (Firestore) — the right home for the VW flag, unlike per-user view prefs.

### Integration Points
- Removing `teamTags` ripples through: `song.ts` type → `SongTable`/`SongSlotPicker` rendering → `songSearch.ts` (`team:` prefix) → `suggestions.ts` (filter + nudge) → `pcSongImport.ts` (mapping) → tag-filter union in `songs.ts` → tests in each. This is the widest-blast-radius decision; plan it as a coordinated wave.
- VW mode is a single church-level flag read by many surfaces (song list, picker, service editor, search, filters) — introduce one source of truth (settings store getter) that each surface consumes to gate VW UI (D-16).

<specifics>
## Specific Ideas

- User's mental model (verbatim intent): "I want the concept of Orchestra and Choir that determine if those teams are ON for a given Sunday. But this is separate from SONGS tagged as Orchestra. Songs should not be linked intrinsically to a given team. Songs should only have 1 set of tags."
- Soft nudge, not hard filter: "The only team that dictates what songs we choose from are the days when the Orchestra team is playing... let's keep a soft nudge. It's possible we'll have big choir numbers we can tag as Choir... Soft only."
- VW is an optional enhancement: "I understand not everyone will want to use this... We should make it an optional package... Something in the settings we can turn on or off. When it's off, we just visually hide any indicators and categories related to vertical worship."
- Source material for the explainer (the user's pasted description of Vertical Worship / Vertical Church Band — "Gather, Testimony, Arrive", horizontal→vertical, "end on a 5/3", songs with no personal pronouns = pure ascription). Summarize this to the app's 1-2-3, do NOT adopt the 1-5 scale (D-18).

</specifics>
