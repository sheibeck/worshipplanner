# Phase 16.1: Song list tags & columns customization - Research

**Researched:** 2026-07-12
**Domain:** Vue 3 + Pinia + Firebase/Firestore SPA — data-model consolidation (teamTags→tags fold), UI column customization, church-level feature flag
**Confidence:** HIGH (all findings are direct code reads with file:line evidence; no external library research was needed — this phase touches only first-party code)

## Summary

This phase is a pure application-code refactor with no new dependencies. Every research question was answerable by reading the existing codebase — there is no framework/library ambiguity. The four sub-changes (R-01..R-04 / D-01..D-18) touch, respectively: (1) a genuinely wide blast radius across 12+ non-test source files and 9 test files for the `teamTags` fold, (2) a well-understood localStorage persistence template already in `songs.ts` (Phase 12's `tagFilterStorageKey`/`persistTagFilter`/`hydrateTagFilter` trio) that can be copied almost verbatim for column-visibility, (3) a `SongTable.vue` inline-tag-edit pattern (`openInlineEdit`/`commitInlineTag`/`removeUserTag`) that can be generalized to a second field (themes), and (4) a **one-time (non-reactive) Firestore read** of `organizations/{orgId}` in `auth.ts`'s `loadOrgContext()` that is the exact, and only, home for a new church-level `vwModeEnabled` boolean — proven by three prior examples (`orgName`, `orgSlug`, `pcAppId`/`pcSecret`) that already follow this precise read/write/mirror pattern.

The single most consequential finding not called out in CONTEXT.md: **`SongSlideOver.vue` (the Add/Edit Song form drawer) and `ArrangementAccordion.vue` (its nested per-arrangement editor) are major, previously-unlisted `teamTags` blast-radius sites.** `SongSlideOver.vue` has its own dedicated "Team Tags" toggle-button UI (`PREDEFINED_TAGS = ['Choir','Orchestra','Hymn']`) and its own "VW Category" 1/2/3 selector that CONTEXT's canonical-refs list does not mention. Both must be addressed for D-01 (fold) and, arguably, D-16 (VW-mode gating) to be complete.

Also notable: today's `upsertSongs()` merge logic gives `tags` and `hidden` full import-preservation (never clobbered by re-import) but gives **`teamTags` zero preservation** — it flows straight through the `...restIncoming` spread and gets silently overwritten on every PC re-import. Folding `teamTags` into `tags` therefore *upgrades* team-tag data to get free import-safety it never had before — a positive side effect of D-01, not a risk.

**Primary recommendation:** Treat the `teamTags` fold (D-01) as its own coordinated wave (data model → store → 6+ UI surfaces → suggestions.ts → songSearch.ts → pcSongImport.ts → csvImport.ts → tests), separate from the additive Themes-column/column-cog work (R-02/R-03) and the additive VW-toggle/explainer work (R-04), since the fold is a breaking rename across nearly every Song-touching file while the other two are pure additions.

## Common Pitfalls

### Pitfall 1: Tailwind v4 static-class-map convention will silently break new dynamic classes
**What goes wrong:** Both `TeamTagPill.vue` and `SongBadge.vue` use a hardcoded `Record<..., string>` "static class map" (e.g. `variantClasses`, `badgeClasses`) instead of building Tailwind class strings dynamically (e.g. `` `bg-${color}-900/50` ``), with an explicit code comment: "Static map prevents Tailwind v4 purge of dynamic class strings."
**Why it happens:** Tailwind v4's JIT scanner only picks up class names it can find as complete literal strings at build time; template-literal-interpolated class names are invisible to it and get purged from the production CSS.
**How to avoid:** Any new UI this phase adds (column-cog toggle states, VW-mode toggle switch, new Themes-pill styling) MUST use the same static-map-of-full-class-strings pattern, never string interpolation for Tailwind classes.
**Warning signs:** A new toggle/badge renders correctly in dev (`vite dev` doesn't purge) but loses all styling after `npm run build`.

### Pitfall 2: `loadOrgContext()` is a one-time `getDoc`, not a live `onSnapshot` — settings changes don't propagate across tabs/sessions without a manual mirror-write
**What goes wrong:** `auth.ts`'s `orgName`, `orgSlug`, `pcAppId`, `pcSecret` are populated once via `getDoc` at login (`loadOrgContext`, lines 76-102) and never re-synced from Firestore afterward. Every existing settings-write flow (`onSave`, `onSaveSlug`, `onSavePcCredentials` in `SettingsView.vue`) compensates by manually reassigning the `authStore` ref right after the `updateDoc` succeeds (e.g. `authStore.orgName = trimmed`).
**Why it happens:** Only the `members/{uid}` subdoc uses `onSnapshot` (for live role changes); the parent org doc read was implemented as a one-shot fetch.
**How to avoid:** The new `vwModeEnabled` field MUST follow the identical mirror-write pattern: after the Settings toggle's `updateDoc` succeeds, immediately set `authStore.vwModeEnabled = newValue` in the same function — do not rely on Firestore pushing the update back automatically.
**Warning signs:** Toggling VW mode in Settings appears to work in the tab that changed it, but a second open tab/session for the same org still shows VW indicators until a full reload.

### Pitfall 3: `upsertSongs()`'s destructuring means silently-included fields get overwritten, silently-excluded fields get preserved
**What goes wrong:** The merge logic explicitly destructures out `vwTypes`, `hidden`, `primaryArrangementId`, `tags`, `themes` to special-case them, then spreads everything else via `...restIncoming` (songs.ts lines 259-266) — meaning `teamTags` today is NOT preserved (it flows through the spread and gets clobbered every re-import). After the fold, this exact pattern must be re-checked: **if the fold simply removes `teamTags` from the type without changing this destructure, nothing breaks — but if a planner mistakenly leaves `Song.tags` out of the special-case destructure while adding new theme-removal-tracking fields, those new fields will silently be overwritten on next import.**
**Why it happens:** The destructure list is a manually-maintained allowlist; it's easy to add a new preserved field to the `Song` type and forget to add it to this destructure.
**How to avoid:** Any new field this phase adds to `Song` that needs import-preservation semantics (e.g. a hypothetical `removedThemes` array for D-14 option B) must be added to the same destructure block in `upsertSongs`, exactly like `tags`/`themes`/`hidden` already are.
**Warning signs:** A field that should survive re-import mysteriously resets after the next PC sync.

### Pitfall 4: The picker's `rankSongsForSlot` and `SongSlotPicker`'s own tag-union both independently reference the three-field union — missing one during the fold breaks only one surface, not both
**What goes wrong:** The `teamTags ∪ themes ∪ tags` union appears in at least 4 independent places with slightly different code: `songs.ts` `filteredSongs` (lines 55-71), `SongSlotPicker.vue` `availableTags`/`tagFilteredSongs` (lines 251-287), `SongsView.vue` `availableUserTags` (lines 223-234), and `suggestions.ts`'s hard filter (lines 42-49, to be removed per D-03). Fixing one and missing another produces an inconsistency where, e.g., the Songs panel's tag checklist offers a tag that the picker's checklist doesn't (or vice versa) — this exact class of bug happened in Phase 12 (see STATE.md's Phase 12 gap-closure plans 12-06/12-07).
**Why it happens:** No shared helper computes "all distinct tags across songs" — each surface reimplements the scan locally.
**How to avoid:** Grep for `teamTags ?? \[\]` / `.teamTags` across the whole `src/` tree (not just the files CONTEXT.md names) before considering the fold done — this research found 21 files matching `teamTags` (see `## Sources` — verified via `Grep`), several of which (`SongSlideOver.vue`, `ArrangementAccordion.vue`, `csvImport.ts`) are not in CONTEXT.md's canonical-refs list.
**Warning signs:** Tag checklist popover shows a different tag set in the Songs panel vs. the service-plan picker.

// Source: src/components/SongTable.vue lines 396-426
async function commitInlineTag(song: Song) {
  const tag = inlineTagInput.value.trim()
  if (!tag) { cancelInlineTag(); return }
  const existingTags = song.tags ?? []
  if (!existingTags.includes(tag)) {
    await songStore.updateSong(song.id, { tags: [...existingTags, tag] })
  }
  cancelInlineTag()
}
async function removeUserTag(song: Song, tag: string) {
  await songStore.updateSong(song.id, { tags: (song.tags ?? []).filter((t) => t !== tag) })
}
// For Themes: identical shape, targeting song.themes / { themes: [...] } instead.

const updateData: Record<string, unknown> = {

## Open Questions

1. **Should `Arrangement.teamTags` (a separate field from `Song.teamTags`) also fold, or stay as-is?**
   - What we know: `Arrangement.teamTags` is a genuinely different field, populated only by the (dead-code, per STATE.md Phase 09 note) `CsvImportModal`/`csvImport.ts` path — `pcSongImport.ts`'s `mapPcSongToUpsert` always sets `teamTags: []` on every mapped arrangement (line 103), meaning live PC-import data never populates it. It IS actively editable today via `ArrangementAccordion.vue`'s per-arrangement "Team Tags" toggle buttons, which feed `SongSlideOver.vue`'s `availableTags` scan.
   - What's unclear: CONTEXT.md explicitly defers this to "planner decides" (canonical_refs, Data Model section) — this research surfaces the evidence but does not resolve it.
   - Recommendation: Since `Arrangement.teamTags` is functionally dead weight from PC import (always `[]`) and only reachable through the dead-code CSV path plus one live manual-toggle UI, the lowest-risk choice is to leave the `Arrangement.teamTags` field and its `ArrangementAccordion.vue` UI untouched in this phase — but flag to the user that this leaves a second, smaller "team tag" concept alive at the arrangement level, which may look inconsistent with the "songs have exactly ONE set of tags" framing of D-01. Worth a one-line confirmation during planning.
