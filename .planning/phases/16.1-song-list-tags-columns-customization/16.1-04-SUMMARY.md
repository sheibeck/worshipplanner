# Phase 16.1 Plan 04: Column-Visibility Store State Summary

**columnVisibility Pinia state (category/key/ccli/lastUsed/tags/themes, all-visible default) with toggleColumn/resetColumns actions and per-user/org localStorage persist+hydrate mirroring the existing tagFilter trio.**

## Accomplishments
- Added `columnVisibility` ref to `useSongStore` defaulting all six toggleable columns to visible, with no `title` key (Title is never toggleable per D-08)
- Added `toggleColumn(col)` and `resetColumns()` actions, both reassigning a new object to preserve Vue reactivity
- Added `columnStorageKey()`/`persistColumnVisibility()`/`hydrateColumnVisibility()` mirroring the `tagFilterStorageKey`/`persistTagFilter`/`hydrateTagFilter` trio verbatim, keyed on `wp:songTableColumns:v1:${org}:${uid}`
- Wired `hydrateColumnVisibility()` into `subscribe()` immediately after the existing `hydrateTagFilter()` call
- Hydrate merges the saved payload over the default map so a future new column key defaults visible even against an older saved payload missing that key

## Decisions Made
- Defined `DEFAULT_COLUMN_VISIBILITY` as a single source of truth used by the initial ref, `resetColumns()`, and both hydrate fallback paths (missing key / corrupt payload) to avoid the default map drifting out of sync across call sites.
- Task 1 and Task 2 were executed as a single TDD RED→GREEN cycle (one test commit, one feat commit) rather than two separate cycles, since Task 2's persist/hydrate logic has no independent meaning without Task 1's `columnVisibility` ref — splitting them would have required a throwaway intermediate implementation.
